Doctor Agent & CureBot — Clinical AI for SOAP Workflows

Clinical AI · LLM Agents · RAG (Milvus) · ICD-10/CPT/GPI · Guideline Compliance · Python

Doctor Agent and CureBot are production-oriented clinical agents that draft complete SOAP notes and support decision-making. They transform the patient–doctor conversation and uploaded context into an Assessment (ranked diagnoses with ICD-10) and a structured Plan (procedures with CPT, medications with GPI, referrals, education, follow-up). A compliance pass aligns the result with provider guidelines, reducing rework and improving documentation quality.

1) Problem & Objectives

Goal: Build a physician-style agent that produces reliable Assessment and Plan grounded in prior notes and guidelines, with correct ICD, CPT, and GPI mappings for interoperability.

  • Generate A (diagnoses + ICD-10) and P (CPT/GPI, education, follow-up) from S + O.
  • Ground reasoning with RAG over provider-specific note collections and guideline packs.
  • Enforce compliance before final SOAP assembly.

2) SOAP Explained (in Words)

  • Subjective (S): Patient narrative — symptoms, history, and context captured from conversation and forms.
  • Objective (O): Measurable findings — vitals, exam, lab or imaging summaries, and device data.
  • Assessment (A): Clinician reasoning — likely diagnoses and differentials, prioritized and coded with ICD-10.
  • Plan (P): Next steps — procedures (CPT), medications (GPI with strength/dose/days), referrals, education, and follow-up.

The system composes A and P from S+O, then renders a final SOAP note. This mirrors how clinicians think (evidence → inference → action) and increases reliability and interpretability of results.

3) End-to-End Flow

Input S+O → Retrieve similar prior notes and guideline snippets (RAG) → Produce Assessment → Generate Plan → Apply compliance → Render SOAP.

4) Data Platform (Milvus RAG)

  • Provider-scoped Milvus collections with embedded notes (MX-Bai embeddings) for semantic retrieval.
  • Top-K note retrieval (10–20 per query) with frequency re-weighting to capture provider-specific ICD patterns.
  • Optional LLM re-ranking on retrieved ICD descriptions enhances precision and context match for Assessment.

5) Plan Agent (A → P with CPT/GPI)

Using confirmed diagnoses, the Plan Agent retrieves similar treatment patterns and generates a comprehensive Plan: Reasoning, Therapy/Plan actions, Procedures (CPT), Medications (GPI with full drug metadata), Referrals, Education, and Follow-Up.

6) Evaluation Highlights

Initial runs using Subjective-only inputs missed codes requiring objective exam context (e.g., Z23, I10, Z11.*). Adding Objective data significantly improved accuracy in Assessment and Plan. Remaining errors stemmed from provider-specific templates and differential ambiguities.

7) CureBot (Conversational Surface)

CureBot is a conversational interface that intelligently routes user intent: Assessment Agent for diagnosis reasoning, Medical QA via CureDoc guideline packs, or Plan Generation. If no path applies, it classifies the request as out-of-scope to avoid hallucinations. A curated QA dataset improves reliability.

8) Why This Design Works

  • Clinician-like flow: S+O → A → P mirrors how doctors reason, reducing brittleness and improving interpretability.
  • Stable coding: Separating diagnosis naming from ICD assignment increases reproducibility and code accuracy.
  • Guideline compliance: The compliance agent enforces adherence to dosing, contraindication, and follow-up guidelines.
  • Localized intelligence: Provider-scoped Milvus RAG captures real-world practice nuances rather than generic textbook behavior.
