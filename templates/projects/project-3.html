HPI Agent — History of Present Illness

Clinical NLP · LLM Multi-Agent · SOAP Notes · OLDCARTS · Python · MongoDB

HPI Agent automates the Subjective portion of clinical notes by turning a patient’s Chief Complaint into a complete, accurate, and concise History of Present Illness (HPI). The agent conducts a guided conversation (OLDCARTS), captures other histories (medical, family, social, drug, surgical) and a brief review of systems (ROS), and returns a well-structured HPI paragraph set that drops directly into a SOAP note.

1) Problem & Goals

Goal: Reduce provider documentation time while improving completeness. Given a Chief Complaint (e.g., “ear pain”), ask the right follow-ups, gather context from prior notes, and generate a structured HPI with pertinent positives/negatives and a brief ROS.

2) Functional Requirements

  • Accept a chief complaint and optional prior-note summary.
  • Drive an adaptive question flow based on OLDCARTS and complaint-specific guidelines.
  • Capture and normalize other histories (PMH, PSH, FHx, SHx, meds/allergies) when relevant.
  • Produce HPI + mini-ROS in a constrained schema (headings, short paragraphs, bullet negatives).
  • Persist conversation state and outputs for reuse and auditing.

3) Architecture

4) Approach & Components

4.1 Questioning Agent

  • Guided by OLDCARTS and complaint-specific prompts (ENT, CV, Resp, GI, Neuro, etc.).
  • Asks focused follow-ups, avoids redundancy, keeps patient-friendly phrasing.
  • Extracts normalized fields (onset, duration, severity scale, aggravating/relieving factors).

4.2 Compliance Agent

  • Checks relevance & sufficiency; flags missing OLDCARTS details and key rule-outs.
  • Prevents drift (off-topic chatter) and reinspects length/clarity before generation.

4.3 Note Generator

  • Deterministic schema: short HPI paragraphs + mini-ROS with positives/negatives.
  • De-identification pass, number normalization (e.g., pain 0-10), tense/style consistency.

4.4 Record-Keeping

  • MongoDB stores conversation turns, structured fields, and prior-note summaries.
  • Enables quick follow-up visits and provider review with change tracking.

5) Data & Prompting

  • Guidelines: complaint/system-specific prompt snippets (e.g., red flags for chest pain, ENT differentials for otalgia).
  • Few-shot library: curated mini-examples per complaint to anchor tone/format.
  • Constraints: JSON slot schema during interaction; natural prose only at final render.
  • Safety: block PHI echo, limit medical advice beyond documentation intent.

6) Output Schema (HPI + ROS)

  • Chief Complaint concise statement (patient words when available).
  • HPI 2–4 paragraphs covering OLDCARTS & context (baseline, prior episodes).
  • Pertinent Positives/Negatives inline; objective vitals deferred to “O”.
  • Mini-ROS bullet positives/negatives in the most relevant systems.

7) Evaluation

  • Expert review loop: rubric on completeness, accuracy, coherence, brevity.
  • Automated checks: OLDCARTS coverage %, red-flag capture %, style conformance.
  • User feedback: in-app thumbs/notes to improve examples and prompts.

8) Example (Ear Pain)

Chief Complaint: “Left ear pain for three days.”

HPI (excerpt): Patient reports constant, dull left otalgia for 3 days, onset gradual, worsened by chewing; relief minimal with OTC acetaminophen. Denies fever, otorrhea, hearing loss, recent swimming, or recent air travel. Prior similar episode 1 year ago, resolved with eardrops.

Mini-ROS: ENT: +otalgia; −fever, −otorrhea, −hearing loss. Resp: −cough, −SOB. Neuro: −headache, −vertigo.

9) Engineering Notes

  • Functional boundaries and message-passing keep agents modular and replaceable.
  • Config-driven prompts enable quick expansion to new complaints/specialties.
  • Strict output schema reduces post-edit time and improves EHR paste-ins.
  • Telemetry (latency, token usage, coverage %) supports cost/quality tuning.
